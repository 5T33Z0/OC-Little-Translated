## Introduction

By default, OpenCore displays a generic hard drive icon for Linux (and other) installations in the Boot Picker:

<img width="794" height="343" alt="Generic_Icon" src="https://github.com/user-attachments/assets/1fcfcabc-068b-4a9c-93c3-e2f0f6e26528" />

This can be customized so that OpenCore displays a distribution-specific icon instead. In this example, Fedora is used:

<img width="737" height="305" alt="OC_Bootpicker_00" src="https://github.com/user-attachments/assets/43d00ce8-4549-4743-afb4-6ba7aa8ce1a2" />

OpenCore **0.8.8** introduced three hidden files that together give you full control over how each boot entry appears in the picker:

| File | Controls |
|---|---|
| `.contentFlavour` | Which icon and audio description is shown |
| `.contentVisibility` | Whether and how the entry appears |
| `.disk_label` / `.disk_label_2x` | The text label rendered beneath the icon |

In current OpenCore releases (1.0.x), `.contentFlavour` and `.contentVisibility` are included by default in the OpenCore package under `X64/EFI/BOOT` and `X64/EFI/OC`. They may still be missing if you upgraded from an older version without replacing those folders, or if your EFI was generated by an older third-party tool.

## Prerequisites

Each feature requires the corresponding flag to be set in `Misc ‚Üí Boot ‚Üí PickerAttributes` in `config.plist`:

| Feature | Flag | Bitmask |
|---|---|---|
| Custom icons (`.contentFlavour`) | `OC_ATTR_USE_FLAVOUR_ICON` | `0x0010` |
| Disk labels (`.disk_label`) | `OC_ATTR_USE_DISK_LABEL_FILE` | `0x0002` |
| Custom icons (theme) | `OC_ATTR_USE_VOLUME_ICON` | `0x0001` |

A common combined value for all three is `0x0013` (19 decimal). Use an OpenCore configurator or calculator to generate the correct bitmask for your setup.

## Controlling the icon: `.contentFlavour`

### What it does

`.contentFlavour` tells OpenCanopy which icon and audio description to associate with a boot entry. Without it, OpenCore falls back to generic detection based on the bootloader filename, which often results in the generic hard drive icon.

OpenCore processes this file only if no explicit flavour is enforced via `config.plist`. If a flavour is set manually in `Misc ‚Üí Entries`, the file is ignored for that entry.

### Configuration

The file contains a single flavour identifier string, for example:

```
Fedora
```

or a colon-separated fallback chain, which OpenCore resolves left to right until a matching icon is found:

```
Fedora:Linux
```

The complete list of available flavour identifiers is maintained by Acidanthera:
https://github.com/acidanthera/OpenCorePkg/blob/master/Docs/Flavours.md

### Placement

Place `.contentFlavour` in the same EFI subfolder as the bootloader it describes, next to the `.efi` file:

```
EFI/
‚îú‚îÄ‚îÄ Microsoft/
‚îÇ   ‚îî‚îÄ‚îÄ Boot/
‚îÇ       ‚îú‚îÄ‚îÄ bootmgfw.efi
‚îÇ       ‚îî‚îÄ‚îÄ .contentFlavour     ‚Üê contains: Windows
‚îú‚îÄ‚îÄ fedora/
‚îÇ   ‚îú‚îÄ‚îÄ shim.efi
‚îÇ   ‚îî‚îÄ‚îÄ .contentFlavour         ‚Üê contains: Fedora:Linux
‚îî‚îÄ‚îÄ OC/
    ‚îî‚îÄ‚îÄ .contentFlavour         ‚Üê contains: OpenCore
```

## Controlling visibility: `.contentVisibility`

### What it does

`.contentVisibility` controls whether a boot entry appears in the picker at all. It replaces many legacy use cases that previously required complex `ScanPolicy` calculations or `CustomEntries` workarounds.

### Options

The file contains a single ASCII word with no trailing spaces or line breaks:

| Value | Effect |
|---|---|
| `Enabled` | Entry is shown normally |
| `Disabled` | Entry is hidden from the picker |
| `Auxiliary` | Entry is hidden by default; shown after pressing `Space` |

`Auxiliary` is commonly used for permanently attached USB devices that should not appear on every boot.

### Placement

Place `.contentVisibility` next to the bootloader you want to control.

**Recommended locations:**

```
/System/Volumes/Preboot/{GUID}/System/Library/CoreServices/.contentVisibility
/System/Volumes/Preboot/.contentVisibility
/EFI/BOOT/.contentVisibility    ‚Üê for USB boot media
```

**Supported but not recommended:**

```
/Volumes/{ESP}/.contentVisibility
```

> [!CAUTION]
>
> Placing `.contentVisibility` at the ESP root affects **all** entries on that partition and can unintentionally hide everything.

**Examples by use case:**

- Hide the OpenCore EFI entry ‚Üí place in `EFI/OC/BOOT/`
- Hide Windows ‚Üí place in `EFI/Microsoft/Boot/`
- Hide a Linux distro ‚Üí place in its respective subfolder next to its bootloader

### InstanceIdentifier

OpenCore **0.9.4+** introduced InstanceIdentifier matching, allowing `.contentVisibility` rules to apply only to specific OpenCore instances. This is useful when multiple OpenCore installations exist on the same machine.

The file can contain instance-specific rules:

```
OCA:Disabled
```

or multiple rules, one per line:

```
OCA:Disabled
OCB:Auxiliary
```

If no instance prefix is used, the rule applies globally to all instances. For details, refer to the OpenCore Configuration PDF, section **8.1.1 ‚Äì Boot Algorithm**.

## Controlling the label: `.disk_label` and `.disk_label_2x`

### What they are

`.disk_label` and `.disk_label_2x` are binary bitmap files that OpenCanopy renders as the text label beneath each boot entry icon. `.disk_label` is the 1x version; `.disk_label_2x` is for HiDPI/Retina displays. The label string is rasterized by the tool using Apple's original `BLGenerateOFLabel` logic and rendered in OpenCanopy's built-in font.

> [!NOTE]
>
> `PickerAttributes` must have `0x0002` set for disk labels to display. See [Prerequisites](#prerequisites).

### Placement

Place both files in the EFI subfolder of the OS they label, next to its bootloader:

```
EFI/
‚îú‚îÄ‚îÄ BOOT/
‚îÇ   ‚îú‚îÄ‚îÄ BOOTx64.efi
‚îÇ   ‚îú‚îÄ‚îÄ .disk_label             ‚Üê OpenCore / fallback label
‚îÇ   ‚îî‚îÄ‚îÄ .disk_label_2x
‚îú‚îÄ‚îÄ Microsoft/
‚îÇ   ‚îî‚îÄ‚îÄ Boot/
‚îÇ       ‚îú‚îÄ‚îÄ bootmgfw.efi
‚îÇ       ‚îú‚îÄ‚îÄ .disk_label         ‚Üê Windows label
‚îÇ       ‚îî‚îÄ‚îÄ .disk_label_2x
‚îú‚îÄ‚îÄ fedora/
‚îÇ   ‚îú‚îÄ‚îÄ shim.efi
‚îÇ   ‚îú‚îÄ‚îÄ .disk_label             ‚Üê Fedora / Bazzite label
‚îÇ   ‚îî‚îÄ‚îÄ .disk_label_2x
‚îî‚îÄ‚îÄ OC/
    ‚îú‚îÄ‚îÄ OpenCore.efi
    ‚îú‚îÄ‚îÄ .disk_label             ‚Üê optional OpenCore label
    ‚îî‚îÄ‚îÄ .disk_label_2x
```

> [!NOTE]
>
> Both files are hidden (dot-prefixed). On macOS, press `‚åò + Shift + .` in Finder to toggle hidden file visibility. For manual `Misc ‚Üí Entries` entries, place the files in whichever folder you specified as the entry path.

### Method 1 ‚Äî OpenCore's `disklabel` utility (macOS)

The official tool ships with every OpenCorePkg release under `Utilities/disklabel/disklabel`.

> [!CAUTION]
>
> macOS includes its own unrelated `disklabel` binary at `/sbin/disklabel`. Always invoke the OC version via `./disklabel` or its full path ‚Äî never by name alone.

1. Mount the EFI partition and navigate to the target subfolder:

   ```bash
   sudo diskutil mount /dev/diskXsY
   cd /Volumes/EFI/EFI/Microsoft/Boot   # adjust per OS
   ```

2. Run the OC `disklabel` binary by full path:

   ```bash
   sudo /path/to/OpenCorePkg/Utilities/disklabel/disklabel -e "Windows" .disk_label .disk_label_2x
   ```

3. Repeat for each OS subfolder:

   ```bash
   cd /Volumes/EFI/EFI/fedora
   sudo /path/to/disklabel -e "Bazzite" .disk_label .disk_label_2x
   ```

Spaces in label names are fine. Re-running the command overwrites existing files.

**Full usage reference:**

```
disklabel -d .disk_label image.ppm                   # decode: label ‚Üí PPM image
disklabel -e "Label" .disk_label .disk_label_2x      # encode from text string
disklabel -bgra "Label" .disk_label .disk_label_2x   # encode, BGRA pixel order
```

### Method 2 ‚Äî `bless` command (macOS, simpler)

```bash
sudo bless --folder /Volumes/EFI/EFI/Microsoft/Boot --label "Windows"
sudo bless --folder /Volumes/EFI/EFI/fedora --label "Bazzite"
```

Creates both `.disk_label` and `.disk_label_2x` in one shot. No OC binary required.

### Method 3 ‚Äî Linux / cross-platform (build from source)

The `disklabel.c` source compiles cleanly on Linux with gcc:

```bash
git clone https://github.com/acidanthera/OpenCorePkg.git
cd OpenCorePkg/Utilities/disklabel
gcc -o disklabel disklabel.c
```

Then run it for each OS folder on the mounted EFI partition:

```bash
sudo ./disklabel -e "Windows" /mnt/efi/EFI/Microsoft/Boot/.disk_label \
                              /mnt/efi/EFI/Microsoft/Boot/.disk_label_2x

sudo ./disklabel -e "Bazzite" /mnt/efi/EFI/fedora/.disk_label \
                              /mnt/efi/EFI/fedora/.disk_label_2x
```

> [!TIP]
>
> On Aurora or any other immutable Fedora-based distro, Method 3 is the recommended approach since neither `bless` nor the prebuilt macOS OC binary will run.

## Deploying the files

If `.contentFlavour` or `.contentVisibility` are missing from your EFI, copy them from the official OpenCore package:

1. Download the latest OpenCore release from https://github.com/acidanthera/OpenCorePkg/releases
2. Extract the package
3. In Finder, press `‚åò + Shift + .` to show hidden files
4. Navigate to `OpenCore-1.0.X-RELEASE/X64/EFI/BOOT` or `.../X64/EFI/OC`
5. Copy `.contentFlavour` and `.contentVisibility` to the corresponding folder on your EFI partition
6. Edit each file's content to match the OS in that folder (see sections above)
7. Press `‚åò + Shift + .` again to re-hide hidden files

> [!NOTE]
>
> In modern OpenCore versions (1.0.x), these files are present by default and normally only need to be **relocated or edited**, not created from scratch.
